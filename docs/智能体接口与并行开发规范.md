# AI驱动广告植入系统：智能体协同与API接口开发规范

本文档旨在为8人开发团队提供清晰的接口定义、数据流转规则与并行开发指南。核心目标是**解耦各Agent的开发工作**，确保即使前置节点未完成，后续节点也能基于标准Mock数据顺利进行。

---

## 1. 核心解耦方案：全局状态字典 (Global State)
本项目采用 Langgraph 进行智能体编排。为了让所有人能立刻开工，我们定义一个标准的**全局状态字典（State Dictionary）**。在Langgraph工作流中，所有的Agent**只负责读取 State 中自己需要的部分，并将计算结果写入 State 中对应的字段**。

### 1.1 全局 State JSON Schema 定义
在并行开发期间，每个后端Agent开发者可以在本地创建一个 `mock_state.json` 文件作为输入。

## 2. 各Agent职责、接口定义与并行开发指南

### 🤖 Agent 1: 数据采集与清洗智能体 (数据源头)
**目标**：爬取小红书帖子，解析并存储为结构化本地文件。
- **输入 (Input)**: `request_info.post_url` (从小红书URL提取Post ID)
- **输出 (Output) 写入State路径**: 
  - `raw_data.post_content` (帖子正文文本)
  - `raw_data.media_paths` (下载到本地的图片/视频绝对路径列表)
  - `raw_data.comments` (清洗后的评论列表)
- **物理落盘规范 (关键)**: 爬虫抓取的内容必须落盘到统一目录 `e:/AIAD/data/raw/{post_id}/`。
  - 图片/视频存在 `media/` 子文件夹。
  - 评论存在 `comments.json`。
- **交付物**: 封装好的Python类 `MediaCrawlerAgent`，包含 `run(url)` 方法。
- **测试标准**: 输入一个有效URL，检查指定目录下是否完整生成了 `media` 文件夹及解析好的JSON。

---

### 🤖 Agent 2: 多模态视觉理解智能体
**目标**：理解图片/视频传递的氛围与场景。
- **并行开发方法**: 无需等Agent 1，自己找2-3张小红书截图放在本地，将路径填入 `mock_state.json` 的 `raw_data.media_paths` 进行开发。
- **输入 (Input)**: `raw_data.media_paths`
- **输出 (Output) 写入State路径**: `vision_analysis`
- **交付物**: 视觉大模型API调用类 `VisionAgent`及视觉Prompt。
- **测试标准**: 传入几张本地测试图片，确保输出的JSON格式完全符合 `vision_analysis` 的 Schema 要求（包含 scene, vibe 等）。

---

### 🤖 Agent 3: 评论区语境与情感智能体
**目标**：分析评论区用户痛点与流行语境。
- **并行开发方法**: 自己编造或复制50条典型的小红书评论，写死在 `mock_state.json` 的 `raw_data.comments` 里。
- **输入 (Input)**: `raw_data.comments`
- **输出 (Output) 写入State路径**: `nlp_analysis`
- **交付物**: LLM调用类 `NLPAgent` 及情感分析Prompt。
- **测试标准**: 传入编造的带明确痛点（如“求链接”）的评论，检查模型能否准确提取出 `pain_points` 和 `language_style`。

---

### 🤖 Agent 4: 记忆与检索智能体 (RAG)
**目标**：提供高质量对标文案。
- **并行开发方法**: 提前构建一个包含50-100条优秀广告文案的Chroma向量库（用CSV导入）。根据预设的 `vision_analysis` 和 `nlp_analysis` 假数据进行查询。
- **输入 (Input)**: `vision_analysis` (作为场景检索词), `nlp_analysis` (作为情绪检索词)
- **输出 (Output) 写入State路径**: `rag_references` (Top-K 字符串列表)
- **交付物**: `ChromaDBManager` 类及检索 Agent 逻辑。
- **测试标准**: 传入特定场景词（如“露营”），断言检索出的文案列表中必须包含相关高频词。

---

### 🤖 Agent 5: 核心文案生成智能体
**目标**：综合所有上下文生成广告。
- **并行开发方法**: 完全依赖 `mock_state.json` 中的前置数据（场景、评论、产品信息、RAG参考）来精调核心Prompt。
- **输入 (Input)**: `request_info.product_info`, `vision_analysis`, `nlp_analysis`, `rag_references`
- **输出 (Output) 写入State路径**: `final_ads` (包含不同风格的广告字典列表)
- **交付物**: 核心 `CopywriterAgent` 类和极度调优的 System Prompt。
- **测试标准**: 给定Mock数据，生成3条以上风格迥异但融入自然的广告评论，人工审查其“广告味”是否足够隐蔽。

---

### 🤖 Agent 6: 工作流编排与后端API (Langgraph)
**目标**：把上面5个人的代码串起来，提供给前端。
- **并行开发方法**: 让 1-5 号开发者先提供一个只返回假数据的空类（如 `def run(state): return dummy_data`）。用这些 Dummy Nodes 先把图跑通。
- **输入 (Input)**: 接收前端发来的 HTTP POST 请求 (URL + 产品信息)。
- **输出 (Output)**: 
  - 过程状态输出（SSE或WebSocket）：向前端实时发送 "正在抓取数据...", "正在进行视觉分析..." 等进度。
  - 最终API返回：JSON格式的 `final_ads`。
- **交付物**: FastAPI 后端服务 + `Graph` 编排逻辑。
- **测试标准**: 使用Postman发送请求，能走完完整的 Dummy 图流转，并返回预期的最终 JSON。

---

### 🤖 Agent 7: 场景交互设计 (Figma)
**目标**：决定产品怎么用。
- **并行开发方法**: 不需要管后端怎么写，专注用户体验。基于“输入URL -> 看进度 -> 选文案”的逻辑画图。
- **交付物**: Figma 原型链接，包含状态标注（如：加载中、成功、分析失败的UI样式）。
- **测试标准**: 团队内部走查，确保UI没有让用户迷惑的死胡同操作。

---

### 🤖 Agent 8: 前端开发 (Vue3)
**目标**：实现UI并对接后端。
- **并行开发方法**: 根据 Agent 6 定义的请求格式，在本地用 Mock.js 或直接写死 JSON 来模拟后端 API 返回。
- **输入 (Input)**: Agent 7 的设计稿，用户输入的表单数据。
- **输出 (Output)**: 用户浏览器中展示的页面。
- **交付物**: Vue3 SPA 源码。
- **测试标准**: 在不连接真实后端的情况下，点击“生成”按钮能看到完整的进度条动画，并最终渲染出 Mock 的多条广告卡片。

---

## 3. 日常联调与合并流程
1. **Mock先行**：所有 Agent (1-5) 在第一周必须能够接收一个 `mock_state.json`，并成功返回自己负责更新后的 State 部分。
2. **节点打通**：Agent 6 将各人的代码引入 Langgraph 节点，替换掉原有的 Dummy Nodes。
3. **真实数据跑通**：由 Agent 1 真实爬取一个 URL，将数据流转给 Agent 2-5，观察最终生成的广告质量。
4. **前后端联调**：Agent 8 连接 Agent 6 的接口，完成最终 SaaS 形态的展示。
